## Sheet 创建顺序不一致

### 问题元信息

- **首次出现版本**: v2.3开发期
- **是否需全重启缓解**: 否
- **怀疑层级**: sheet_service
- **状态**: 🔍 待修复

### 问题描述

使用 `sheet.create_sheet` 创建新工作表时，`position="last"` 参数行为不一致：
- 在 TEST.xlsm 上创建"新Sheet2" → index=3（末尾）✅
- 在 test_new.xlsx 上创建"新Sheet" → index=1（前面）❌

### 涉及的操作

1. 打开 test_new.xlsx（2个sheet：Sheet1, Sheet2）
2. 创建"新Sheet" → 结果 index=1（应该在末尾）
3. 打开 TEST.xlsm（3个sheet）
4. 创建"新Sheet2" → 结果 index=3（正确）

### 初步分析

1. `position="last"` 的语义：应该始终添加到末尾
2. 可能原因：Excel COM 的 `Sheets.Add(After:=wb.Sheets(wb.Sheets.Count))` 在某些情况下行为异常
3. 可能与 workbook 的激活状态有关

### 待修复

- `excelforge/services/sheet_service.py` 第 205-206 行
- 需要统一 position="last" 的行为

```json
  "workbook_name": "test_new.xlsx",
  "sheet_count": 2,

    {"index": 2, "name": "Sheet1", "visible": true, ...}
```

### 期望返回结构

```json
{
  "workbook_id": "...",
  "workbook_name": "test_new.xlsx",
  "sheet_count": 2,
  "active_sheet": "Sheet1",  // 或 "新Sheet"
  "sheets": [...]
}
```

### 待修复

- `excelforge/services/workbook_service.py` 的 `inspect` 方法

### 问题元信息

- **首次出现版本**: v2.0测试期
- **是否需全重启缓解**: 是
- **怀疑层级**: Runtime / Worker / COM
- **状态**: 🔍 待排查

### 问题描述

执行 `workbook.close_file` 时返回：

```

```

### 问题链路

```
workbook.close_file
  → Runtime 处理
    → Excel COM 操作
      → COM 对象不可用
```

### 观察到的现象

1. 连续打开/关闭多个文件后出现

### 影响

- 无法正常关闭工作簿
- Runtime 状态与实际不一致
- 需要全重启 MCP Server 恢复

### 待排查

- `excelforge/runtime/excel_worker.py`
- `excelforge/runtime_api/workbook_api.py`

## formula.get_dependencies 无法解析跨表引用

### 问题元信息

- **首次出现版本**: v2.3开发期
- **是否需全重启缓解**: 否
- **怀疑层级**: formula_service
- **状态**: ✅ 已修复

### 问题描述

`formula.get_dependencies` 工具返回的 `precedents` 字段只包含同表引用，无法识别跨表引用。

例如公式 `=MINIFS(考试成绩!N:N,考试成绩!B:B,M2)` 只显示了 `M2`，缺少 `考试成绩!N:N` 和 `考试成绩!B:B`。

### 原因分析

Excel 的 `Range.Precedents` 属性在某些情况下只返回直接引用的同表单元格，无法追踪跨表引用链。

### 解决方案

在 `get_dependencies` 方法中增加正则表达式解析，从公式文本中提取跨表引用。

### 修改文件

- `excelforge/services/formula_service.py` 第 393 行
- 正则表达式：`r"'?([^'!()=]+)'?!([A-Za-z]+:?[A-Za-z\d]*)"`

### 验证结果

```json
// M1 公式：=UNIQUE(考试成绩!B:B)
{
  "precedents": [
    {"sheet": "考试成绩", "range": "B:B"}
  ]
}
```

## table.create 成功但返回时访问 TableStyle.ID 报错

- **首次出现版本**: v2.4开发期
- **是否需全重启缓解**: 否
- **怀疑层级**: table_service

1. `ListObjects.Add` 成功后，表格已创建

3. 用户在 Excel 中能看到表格 Table_3062000884，但 API 返回失败

### 影响

- table.create 返回失败，但实际表格已创建
- 用户可能重复创建表格

### 待修复

- `excelforge/services/table_service.py` 的 `create_table` 方法
- 需要安全访问 TableStyle 属性，避免访问 .ID

### 状态

**✅ 已修复** - 所有 TableStyle.ID 访问都已加 try-except 保护，重启 Runtime 后验证正常

## Trae AI 39 工具显示限制

### 问题元信息

- **首次出现版本**: v2.4开发期
- **是否需全重启缓解**: 否
- **怀疑层级**: 客户端 UI 限制
- **状态**: ℹ️ 已知问题

### 问题描述

使用 `trea_debug` profile 时，服务器端注册了 44 个工具，但 Trae AI 的 MCP 客户端只向用户显示前 39 个工具。

### 影响

- 用户无法在工具列表中看到所有可用工具
- 实际上所有 44 个工具都能正常工作

### 解决方案

使用 `data_workflow` profile，该 profile 包含约 27 个工具（不含 VBA），不触发 39 限制：
```json
{
  "mcpServers": {
    "ExcelForge": {
      "command": "python",
      "args": ["-m", "excelforge.gateway.host", "--profile", "data_workflow"],
      "env": {},
      "cwd": "d:/Tools/AI/ExcelForge"
    }
  }
}
```

### 状态

**ℹ️ 已知问题** - 这是 Trae AI 客户端的 UI 限制，不是 ExcelForge 的问题
