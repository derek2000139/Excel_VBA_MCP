### 1.4 当前开发经验结论

#### 结论 1：Profile 必须按用户场景设计

当前 Profile 本质上是"大 Bundle 直接相加"，不是场景入口。\
用户想完成"基础编辑"，不需要 31 个工具。

#### 结论 2：30 是设计目标，39 是风险线

* `≤ 30`：理想状态，为后续新增工具预留空间

* `31 ~ 39`：可接受，但应有设计说明

* `≥ 40`：受限客户端高风险，不适合作为默认场景 Profile

* 这两个值都不是硬限制，而是设计参考

#### 结论 3：不必要的工具应删除，不应仅隐藏

`vba.execute_inline` 的彻底移除已证明：与现有工具语义重叠的能力应优先删除。

#### 结论 4：客户端兼容应作为开发测试约束

在设计和验证时纳入考虑，但不替代本版主功能目标。

#### 结论 5：框架必须为非 Runtime 工具和多后端预留位

后续 package / OOXML 工具不依赖 Excel Runtime，WPS v3 需要多后端架构。这些在 v2.4 框架设计中需要预留。

***

### 1.5 业务目标

v2.4 的核心目标是：

> 在继续补充新工具与新能力域（table、analysis 扩充、导出能力等）的同时，基于用户使用场景重整 Profile / Bundle / YAML 配置组织，控制单场景工具规模，并在框架层为后续 package / OOXML（v2.5）和 WPS 后端兼容（v3）预留扩展点。

## 4. Bundle 重新设计

### 4.1 设计原则

1. **单一职责**：每个 Bundle 聚焦一个可复用的能力单元
2. **粒度适中**：太粗导致 Profile 膨胀，太细导致维护复杂
3. **依赖最小化**：减少 Bundle 间依赖
4. **核心前置**：每个 Bundle 的核心工具定义在前部

### 4.2 Bundle 定义

所有场景的公共基础。

## 5. Profile 重新设计

### 5.1 设计原则

1. **场景驱动**：Profile 回答"用户要完成什么任务"，不是"内部有哪些 Domain"
2. **预算目标 ≤ 30**：为后续新增工具预留空间；31\~39 可接受但需说明；≥ 40 不适合默认场景 Profile
3. **按需增强**：用户可通过 `--enable-bundle` 按需扩展，不需要塞进默认组合
4. **分类明确**：区分用户 Profile、开发 Profile、全量 Profile

## 6. 新工具设计

### 6.1 新增工具总览

v2.4 计划新增工具按优先级分为三批：

| 批次  | 涉及域                    |  预计新增数 | 说明        |
| --- | ---------------------- | -----: | --------- |

## 7. 框架预留设计

### 7.1 Tool Manifest 扩展

为后续 package（v2.5）和 WPS（v3）预留以下字段：

```yaml
# 示例 manifest
name: pq.get_code
domain: pq
bundle: data
runtime_method: pq.get_code
requires_excel_ready: true

# v2.4 新增预留字段
execution_mode: runtime          # runtime | file_package | batch
backend_requirement: excel_com   # excel_com | file_only | future_wps
supports_batch: false
default_order: 220
recommended_profiles:
  - data_workflow
  - pq_first
maturity: stable
risk_level: medium
```

| 字段                     | 含义       | 当前默认值                         |
| ---------------------- | -------- | ----------------------------- |
| `execution_mode`       | 执行模式     | `runtime`（所有现有工具）             |
| `backend_requirement`  | 后端依赖     | `excel_com`（所有现有工具）           |
| `supports_batch`       | 是否支持批量输入 | `false`                       |
| `default_order`        | 默认注册顺序   | 按 bundle/域内顺序                 |
| `recommended_profiles` | 推荐场景     | 手工标注                          |
| `maturity`             | 成熟度      | `stable` / `beta` / `preview` |
| `risk_level`           | 风险等级     | `low` / `medium` / `high`     |

### 7.2 Host 对非 Runtime 工具的预留

当前所有工具链路：`Host → Runtime → Excel COM`

后续 package 工具链路：`Host → 文件系统直接处理`

v2.4 在 Host 层预留：

* tool\_manifest 中 `execution_mode` 字段

* Host 装配时可识别 `file_package` 类工具

* 这些工具不需要等待 Runtime ready

> v2.4 不需要实现完整的多执行模式分发，只需要让 manifest 结构和 Host 注册逻辑预留得下这个位置。

### 7.3 server.health 的后端能力预留

建议在 `server.health` 返回结构中预留后端能力声明位：

```json
{
  "runtime_status": "running",
  "excel": {
    "ready": true,
    "version": "16.0"
  },
  "backend": {
    "type": "excel_com",
    "capabilities": {
      "vba": true,
      "pq": true,
      "chart": false,
      "package_extract": false
    }
  }
}
```

> v2.4 可以先输出 `backend.type = "excel_com"` 和一个基本 `capabilities` 结构。不需要实现完整能力矩阵。

### 7.4 错误码预留

为后续多后端场景预留错误码语义区分：

* `E423_TOOL_NOT_ENABLED_IN_PROFILE`：工具未在 Profile 中启用

* `E423_TOOL_NOT_SUPPORTED_BY_BACKEND`（预留）：工具在当前后端不支持

v2.4 不需要使用 `E423_TOOL_NOT_SUPPORTED_BY_BACKEND`，但错误码表中应预留。

### 7.5 package / artifact\_export 的 Bundle / Profile 预留

v2.4 在配置层预留（不实现工具）：

### 9.2 预算语义

| 区间        | 含义       | 处理方式              |
| --------- | -------- | ----------------- |
| `≤ 30`    | 理想       | 正常通过              |

| `≥ 40`    | 受限客户端高风险 | 不适合作为默认场景 Profile |

### 9.3 标准验证流程

每个专题开发前至少执行：

```
1. 确认新增工具已进入正确 domain / bundle
2. 确认 profile 解析结果正确
3. 使用 --dump-tools 检查最终工具数
4. 使用 --dump-tools-with-index 检查关键工具顺序
5. 在目标客户端做最小 happy path 验证
6. 再进入业务缺陷排查
```

### 9.4 问题定性流程

```
1. 工具是否在 profile 中声明？      → 否：profile 配置问题
2. list_tools 是否返回该工具？       → 否：bundle 注册问题
3. 客户端是否显示该工具？           → 否：客户端截断问题
4. 工具是否可调用？                 → 否：连通性问题
5. 调用是否返回业务错误？           → 是：功能缺陷
```
